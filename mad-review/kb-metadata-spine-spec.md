# KB Metadata Spine — Migration Spec

## Goal

Consolidate the 6+ top-of-file comment-line conventions accumulated in the KB into one structured frontmatter block, and move the derived fields (subtree-IDs, eventually bootstrap text) onto a standalone refresh script that runs separately from the read-only verifier.

## Convention surface (before)

Per leaf:
```markdown
[↑ Parent](index.md)
<!-- leaf: verbatim -->
<!-- path-stable: referenced from volX as ... -->     (optional)
<!-- claim-quality: id1, id2 -->                       (or no-claim variant)
```

Per index:
```markdown
[↑ Parent](path)
<!-- claim-quality (subtree): id1, id2, ... -->       (hand-edited but logically derived)
```

Plus per-multi-claim leaf: inline `<!-- claim-quality: id ... -->` markers in body (these stay).

## Convention surface (after)

Per file (leaf, leaf-as-index, index, entry-point):

```markdown
[↑ Parent Title](relative/path)

<!-- kb-frontmatter
kind: leaf
claims: [h9aqmt]
-->

# Title
... body ...
```

The frontmatter is a YAML block inside an HTML comment, opened by the literal marker `<!-- kb-frontmatter` and closed by `-->`. Renders invisibly on GitHub, parses cleanly with PyYAML, greppable with `grep -A 10 kb-frontmatter`.

### Field reference

**Required for every file:**
- `kind` — one of `leaf`, `leaf-as-index`, `index`, `entry-point`.

**For `kind: leaf` and `kind: leaf-as-index`:**
- Either `claims: [id1, id2, ...]` OR `no-claim: <reason>` (mutually exclusive, validator enforces).
- Optional `path-stable: <free-form provenance string>`.

**For `kind: index`:**
- `subtree-claims: [id1, id2, ...]` — **derived field**, regenerated by refresh script. Do not hand-edit.
- Optional `bootstrap: true` — present on volume roots and the entry-point. Signals that the file emits the bootstrap blockquote directive (the directive text itself stays in the body and is hand-maintained for now; future refresh enhancement can auto-generate).

**For `kind: entry-point`:**
- `subtree-claims: [id1, id2, ...]` — derived (union of all leaf claims in the KB).
- `bootstrap: true`.

### Inline Tier 2 markers (unchanged)

Multi-claim leaves still carry inline `<!-- claim-quality: <id> ... -->` markers proximal to each specific claim in the body. These are user-placed and do not move into frontmatter.

### What the migration does NOT touch

- `claim-quality.md` files (cross-cutting and per-volume) — they keep their existing entry format. Frontmatter applies only to the leaf/index files that consume the IDs, not to the registers that define them.
- `CLAUDE.md`, `CONVENTIONS.md`, `LIVING_REFERENCE.md`, `README.md` — top-level docs, not KB content.
- Inline Tier 2 markers in leaf bodies — preserved exactly.
- Bootstrap directive blockquotes — preserved (auto-generation deferred to a later phase).

## Three scripts

### 1. `migrate-to-frontmatter.py` (one-shot)

- Reads each .md file in `manuscript/ave-kb/`.
- Detects file kind from existing top-of-file conventions:
  - Line 2 = `<!-- leaf: verbatim -->` AND directory has siblings → `leaf`
  - Line 2 = `<!-- leaf: verbatim -->` AND directory contains only `index.md` → `leaf-as-index` (and the file IS named `index.md`)
  - File named `index.md`, line 2 contains `<!-- claim-quality (subtree):` → `index`
  - File is `entry-point.md` → `entry-point`
- Extracts existing `claims`, `no-claim`, `path-stable`, `subtree-claims` from comment lines.
- Emits the frontmatter block.
- Removes the obsolete comment lines.
- Preserves up-link line, body, Tier 2 inline markers, bootstrap blockquote.
- Idempotent — running it twice is a no-op.
- Files that don't match any expected pattern: SKIP and report; never silently rewrite.

### 2. `refresh-kb-metadata.py` (standing, has side effects)

- Reads frontmatter of every leaf to collect the canonical `(path, claims)` map.
- For each `kind: index` file, computes `subtree-claims` as the sorted union of leaf claims under its directory.
- For `kind: entry-point`, computes the union over the entire KB.
- Rewrites only the `subtree-claims` field of each affected file.
- Idempotent.
- Run via `make refresh-kb-metadata`.

### 3. `check-claim-quality.py` (read-only, no side effects)

- Parses frontmatter of every file (instead of grepping comment lines).
- Same 8 hard-fail checks as today's validator.
- Failure report categorizes each failure as **refresh-fixable** (subtree-claim drift, missing subtree field) or **manual-fix** (orphan refs, multi-claim missing Tier 2, etc.).
- If any refresh-fixable failures present, appends a hint: "Try `make refresh-kb-metadata` first; if anything remains, those are real defects."
- Run via `make verify-claim-quality`.

## Migration safety

- Write the migration script.
- Run on a sample of 4 representative files: leaf, leaf-as-index, leaf with path-stable, index.
- Verify frontmatter parses, content preserved, validator (still in old form) doesn't see broken state.
- Update validator to read frontmatter.
- Run on full KB.
- Run refresh.
- Run validator. Pass.
- Commit.

## Maintenance flow (after)

1. User adds a new leaf with claims: writes the frontmatter block by hand (4-5 lines).
2. User runs `make refresh-kb-metadata` to update subtree-claims on parent indexes.
3. User runs `make verify-claim-quality` to confirm clean.
4. User commits.

The user's edit surface drops from 4-6 separate comment patterns to one structured block; the derived fields are mechanically maintained.
